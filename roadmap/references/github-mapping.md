# Publishing the roadmap to a GitHub Project (v2)

Load this file after you have:
1. A discovery object (Phase 1 complete)
2. A list of 5–10 features grouped into phases (Phase 2 brainstorming complete)

## Tools

Use the `gh` CLI for everything:

1. **`gh` CLI** (`gh project …`, `gh issue …`, `gh label …`, `gh repo …`) for all typed operations. Requires the `project` auth scope for Projects v2 operations. If a command fails with a scope error, surface it and stop — ask the user to run `gh auth refresh -s project` and retry.
2. **Raw GraphQL** via `gh api graphql` only for operations the typed commands don't cover (e.g. adding a single-select option to an existing field).

## Key GitHub Projects (v2) concepts

- A **Project** belongs to a user or org, identified by a **number** (e.g. `1`, `42`) and a **node ID** (e.g. `PVT_kwHO…`).
- **Fields** belong to a project and have a node ID (e.g. `PVTSSF_…` for single-select).
- For single-select fields, each **option** also has its own ID (e.g. `f75ad846`). You can only resolve option IDs by listing fields with `--format json`.
- **Items** are issues, PRs, or draft issues added to the project. Setting a field value requires: item ID, field ID, project ID, and (for single-select) option ID.
- `gh project item-edit` updates **one field at a time**. Plan for N update calls per item where N = number of fields you set.

---

## Step 1 — Set up the Project (and detect mode)

1. List existing projects for the owner:
   ```
   gh project list --owner <owner> --format json
   ```
   Look for a project titled `<Repo name> Roadmap`.

   - **Project exists → incremental mode.** Record its number and node ID. Continue to Step 2 to reconcile fields, then jump to "State analysis (incremental mode)" below before filing anything.
   - **Project does not exist → greenfield mode.** Create it:
     ```
     gh project create --owner <owner> --title "<Repo name> Roadmap" --format json
     ```
     The JSON response includes `number` (human-facing) and `id` (node ID, `PVT_…`). Keep both.

2. Set a short README on the project so users landing on the board know what it is (only if the project had no README):
   ```
   gh project edit <number> --owner <owner> --readme "Roadmap for <repo>. Generated by the roadmap skill. Filter by Phase, Priority, Complexity, Impact, Sprint."
   ```

## Step 2 — Create custom fields

List existing fields first:
```
gh project field-list <number> --owner <owner> --format json
```
The default project ships with `Title`, `Assignees`, `Status`, `Labels`, `Linked pull requests`, `Milestone`, `Repository`, and `Reviewers`. You need to **add** the following (skip any that already exist by name):

| Field name | Data type | Options |
|---|---|---|
| `Phase` | SINGLE_SELECT | `Phase 1 — Foundation`, `Phase 2 — Enhancement`, `Phase 3 — Scale`, `Phase 4 — Future` (include only the phases you actually populated) |
| `Priority` | SINGLE_SELECT | `Must`, `Should`, `Could`, `Won't` |
| `Complexity` | SINGLE_SELECT | `Low`, `Medium`, `High` |
| `Impact` | SINGLE_SELECT | `Low`, `Medium`, `High` |
| `Sprint` | SINGLE_SELECT | `Sprint 1` (greenfield) OR the existing sprint options plus `Sprint N+1` (incremental) |
| `Competitor` | TEXT | — (only if competitor data was provided) |

**Extending options on existing single-select fields (applies to both `Phase` and `Sprint`):**

Adding a single option to an already-created single-select field isn't directly supported by `gh project field-create` (that command creates a whole new field). The reliable path is GraphQL — read the field's current options, append the new one(s), write the full list back:

```
gh api graphql -f query='
  mutation($projectId:ID!, $fieldId:ID!, $options:[ProjectV2SingleSelectFieldOptionInput!]!) {
    updateProjectV2Field(input:{
      projectId:$projectId,
      fieldId:$fieldId,
      singleSelectOptions:$options
    }) { projectV2Field { ... on ProjectV2SingleSelectField { id name options { id name } } } }
  }' \
  -f projectId="$PROJECT_NODE_ID" \
  -f fieldId="$FIELD_NODE_ID" \
  -F options='[{"name":"Phase 1 — Foundation","color":"GRAY"},{"name":"Phase 2 — Enhancement","color":"GRAY"},...,{"name":"Phase 5 — Enterprise","color":"GRAY"}]'
```

After the mutation, **re-run `field-list --format json`** to pick up the new option IDs and refresh your `option_ids` lookup table.

**Sprint field specific rules:**
- Greenfield: create the Sprint field with just `Sprint 1` as its single option.
- Incremental, Sprint field already exists: list its options, find the highest numeric suffix among `Sprint <N>` entries, append a new option `Sprint N+1` using the mutation above.
- Incremental, Sprint field does NOT exist (project was created by a prior skill version): create it with options `Sprint 1, Sprint 2` — `Sprint 1` becomes the backfill target for existing items (see State analysis step) and `Sprint 2` is for the new items.
- Non-numeric sprint names that a human added (e.g. "Hotfix Sprint") are preserved but ignored when computing N+1.

**Phase field specific rules:**
- Greenfield: create the Phase field with one option per phase you actually populated (typically `Phase 1 — Foundation`, `Phase 2 — Enhancement`, `Phase 3 — Scale`, optionally `Phase 4 — Future`).
- Incremental, reusing existing phases: don't touch the Phase field options in this step — the existing options are sufficient.
- Incremental, opening a new phase: see §2.5.4 for when this is justified. The new option is added via the mutation above; naming follows `Phase <N+1> — <Theme>` where N is the highest numeric suffix currently on the Phase field.
- Non-numeric phase names a human added are preserved but ignored when computing N+1.

Create each field:
```
gh project field-create <number> --owner <owner> \
  --name "Priority" \
  --data-type SINGLE_SELECT \
  --single-select-options "Must,Should,Could,Won't" \
  --format json
```

Re-run `field-list --format json` after creation and build this in-memory lookup table — you'll need it constantly in Step 6:

```
field_ids = {
  "Phase":      "PVTSSF_...",
  "Priority":   "PVTSSF_...",
  "Complexity": "PVTSSF_...",
  "Impact":     "PVTSSF_...",
  "Sprint":     "PVTSSF_...",
  "Status":     "PVTSSF_...",   # default field, still needed to set Todo
  "Competitor": "PVTF_...",     # if present
}
option_ids = {
  "Phase": {
    "Phase 1 — Foundation": "abc12345",
    "Phase 2 — Enhancement": "def67890",
    ...
  },
  "Priority": {
    "Must": "...",
    "Should": "...",
    "Could": "...",
    "Won't": "...",
  },
  "Complexity": {"Low": "...", "Medium": "...", "High": "..."},
  "Impact":     {"Low": "...", "Medium": "...", "High": "..."},
  "Sprint":     {"Sprint 1": "...", "Sprint 2": "...", ...},   # updated after adding the new sprint option
  "Status":     {"Todo": "...", "In Progress": "...", "Done": "..."},
}
```

If field creation fails (permission / scope error on a specific field), continue without that field and note the gap in the final summary. Never retry silently.

## Step 2.5 — State analysis (incremental mode only)

Skip this step entirely in greenfield mode.

**Goal**: understand what's on the board before generating new features, so the next sprint complements rather than duplicates existing work.

### 2.5.1 — Pull every item with its field values

```
gh project item-list <number> --owner <owner> --format json --limit 500
```

For each item in the response, extract:
- `content.number`, `content.title`, `content.url` (the underlying issue)
- `fieldValues.nodes` — the Status, Phase, Priority, Sprint, etc. values

Note: `item-list` returns a flattened view and may omit some field values. If you need the complete field matrix, fall back to `gh api graphql` with a `ProjectV2Item` query.

### 2.5.2 — Backfill Sprint field if it was just created

If the Sprint field was newly created in Step 2 (because the project predates Sprint support), tag every existing item with `Sprint 1`:

```
gh project item-edit --id <item-id> --project-id <project-node-id> \
  --field-id <sprint-field-id> --single-select-option-id <sprint-1-option-id>
```

Do this before generating new features so the board is uniformly organized. Batch these updates; they're mechanical.

### 2.5.3 — Compute the state summary

From the item list, compute:

```
state = {
  total_items: <count>,
  by_status: { Todo: N, "In Progress": N, Done: N, null: N },
  by_phase:  { "Phase 1": {todo:N, in_progress:N, done:N}, ... },
  by_sprint: { "Sprint 1": {todo:N, done:N, ...}, ... },
  highest_sprint_number: <int>,       # max N where option name is "Sprint N"
  next_sprint_number:    <int>,       # = highest_sprint_number + 1
  open_items: [                       # Todo + In Progress, for carryover context
    {number, title, phase, sprint, age_days},
    ...
  ],
  stalled_items: [                    # open AND age_days > 60
    {number, title, phase, sprint, age_days},
    ...
  ],
  completion_rate_per_phase: { "Phase 1": 0.8, "Phase 2": 0.2, ... },
}
```

Use the issue `created_at` from `gh issue view <N> --json createdAt` (batch with `--jq` if needed) to compute `age_days`. If that's too expensive for 100+ items, skip the age computation and drop the stalled detection — it's a nice-to-have, not required.

### 2.5.4 — Feed state into feature generation

Before generating the next sprint's features, fold `state` into the context alongside the discovery object. Rules:

1. **Do not regenerate existing open items.** If the discovery object would suggest a feature whose title substantially matches an existing open item, skip it silently.
2. **Prioritize finishing before expanding.** If a phase has `completion_rate < 0.5` and there are open items in it, prefer adding features that unblock or complement those open items over opening a new phase — unless the user explicitly said otherwise.
3. **Respect user steering.** If the user's prompt for this run included hints like "focus on performance this sprint" or "ignore Phase 3 for now", use those as priority weights. Honor them over the default "finish earliest phase first" heuristic.
4. **Size the sprint.** Target 3–7 new features. If fewer natural candidates exist (e.g. the roadmap is nearly complete), file fewer — even a single new feature is valid if it's the right one. Don't pad.
5. **Dependencies can reference existing open items.** The new feature's `dependencies` array can list titles of existing open items from `state.open_items`. They'll be resolved to real issue numbers in Step 7.
6. **Opening a new phase** (reusing existing phases is the default — this is the exception). Decide to open a new `Phase N+1 — <Theme>` *only* if at least one of these conditions is met:
   - **(a) Explicit user ask.** The user's prompt for this run requested a new phase directly ("add a phase for mobile", "start Phase 5 on enterprise features", "open a new phase for observability"). In this case the theme follows the user's wording.
   - **(b) Organic emergence.** All three sub-conditions must hold simultaneously:
     1. At least **2 of the newly generated features** share a coherent theme that doesn't map cleanly onto any existing phase.
     2. The earliest existing phase with open items has `completion_rate >= 0.8` — i.e., you're not abandoning an unfinished earlier phase to start a new one.
     3. The total phase count after adding would be **≤ 6**. Beyond that, the board is harder to navigate than it helps — prefer stuffing the new features into the closest existing phase and flagging it in the summary.

   If (a) or (b) is met, choose a 1–3 word theme name (e.g. `Enterprise`, `Mobile`, `Observability`, `AI-native`). The new phase's number is `max(existing_phase_numbers) + 1`. Record it as `new_phase = {"number": N+1, "name": "<Theme>", "full_option": "Phase N+1 — <Theme>"}` so Step 6 knows to extend the Phase field.

   If neither (a) nor (b) is met, force every new feature into an existing phase — even if the fit is imperfect — and note the awkward fits in the summary so the user can re-triage later.

Carry `state`, `new_phase` (or `None`), `open_items`, `stalled_items`, and `next_sprint_number` forward — Step 6 and Step 8 need them.

## Step 3 — Prepare repo labels (slimmed set)

With custom fields carrying priority, complexity, and impact, labels can stay minimal. Create the following if they don't already exist in the repo (treat "already exists" as success):

| Label | Color | Purpose |
|---|---|---|
| `roadmap` | `#0E8A16` | Marks all issues created by this skill |
| `priority:must` | `#B60205` | Mirrors the Priority field for repo-level filtering |
| `priority:should` | `#D93F0B` | " |
| `priority:could` | `#FBCA04` | " |
| `priority:wont` | `#CCCCCC` | " |
| `status:idea` | `#EDEDED` | Newly created, awaiting triage |
| `competitor-insight` | `#FF7F50` | Only if competitor data was provided |

Skip any label whose creation fails and note it in the final summary — issues still work without them.

## Step 4 — Check for existing work

Before filing anything:

1. List existing open issues (`gh issue list --repo <owner/repo> --state open --limit 100`). Skim titles.
2. For each planned feature, decide whether an open issue already describes it. Use conservative matching — if the titles clearly refer to the same feature, mark that feature as `duplicate_of: #<N>` and don't file a new issue.
3. **Greenfield mode, duplicates found: still add the existing issue to the project and set the fields** in Step 6. The point is a complete board, not a churn of new issues.
4. **Incremental mode: duplicate checking against open items was already done in Step 2.5.4** using the project's item list, which is authoritative. This Step 4 pass is a belt-and-suspenders check against *repo* issues that aren't on the project — if any are found, decide per-issue whether to add them to the project (with best-effort field values) or skip them.

## Step 5 — File issues in dependency order

Order features so every dependency is filed before anything that depends on it. Within a phase, file `must` before `should` before `could`.

For each **non-duplicate** feature, create an issue:

```
gh issue create --repo <owner/repo> \
  --title "<feature title>" \
  --body-file /tmp/issue-body.md \
  --label roadmap,priority:<value>,status:idea[,competitor-insight]
```

Body template (exact):

```markdown
## Summary
<description — 2–3 sentences>

## Rationale
<why this matters for the primary persona. If the feature has competitor insights, include: "Addresses <competitor name>'s pain point: <pain point text>">

## User stories
- <story 1>
- <story 2>

## Acceptance criteria
- [ ] <criterion 1>
- [ ] <criterion 2>
- [ ] <criterion 3>

## Scoring
- **Priority**: <must | should | could | wont>
- **Complexity**: <low | medium | high>
- **Impact**: <low | medium | high>
- **Phase**: <N — phase name>

(Also tracked on the [roadmap project](<project URL>) — these values live as custom fields there.)

## Dependencies
<if no dependencies: "None.">
<otherwise, for each dependency, a placeholder line: "- Depends on <feature title> (issue TBD)">

---
_Generated by the [`roadmap`](https://github.com/hiboute/auto-claude) skill._
```

After creating each issue, record a map `feature_title -> {issue_number, issue_url}`. You need it for:
- Step 6 (adding items to the project)
- Step 7 (back-filling dependency numbers)

## Step 6 — Add issues to the project and set fields

**6.0 — Ensure all required options exist on the Phase and Sprint fields.**

Before setting any field values, make sure every Phase and Sprint option referenced by the new features is present on the respective field. Use the option-extension mutation from Step 2 if anything is missing:

- **Sprint** — the new `Sprint N+1` option should already have been created in Step 2. Verify it exists; if not (e.g., a prior error), add it now.
- **Phase** — if §2.5.4 produced a `new_phase` decision, extend the Phase field's options to include `Phase N+1 — <Theme>`. If `new_phase` is `None`, skip — all features should already map to existing options.

After any extension, re-fetch `field-list --format json` and refresh the `option_ids` lookup. A stale lookup here is the most common cause of silent field-set failures.

For each feature (including duplicates that already had issues):

1. **Add the item** to the project and capture the item node ID:
   ```
   gh project item-add <number> --owner <owner> --url <issue URL> --format json
   ```
   The `.id` in the JSON response is the item ID (`PVTI_…`). Store it keyed by feature title.

2. **Set each field value.** One `item-edit` call per field:

   **Single-select** (Phase, Priority, Complexity, Impact, Sprint, Status):
   ```
   gh project item-edit \
     --id <item-id> \
     --project-id <project-node-id> \
     --field-id <field-id-from-lookup> \
     --single-select-option-id <option-id-from-lookup>
   ```

   **Text** (Competitor, if set):
   ```
   gh project item-edit \
     --id <item-id> \
     --project-id <project-node-id> \
     --field-id <field-id-from-lookup> \
     --text "<competitor name>: <pain point>"
   ```

   **Sprint assignment rules:**
   - Greenfield: tag every new item with `Sprint 1`.
   - Incremental: tag every new item with `Sprint N+1` (the option you added in Step 2). **Do not touch the Sprint value on pre-existing items** — they keep whatever sprint they were in.

   Always set `Status` to the `Todo` option ID so the new items show up in the default board's first column. If `Status` is the only field you successfully wrote, that's still a useful board — the other fields are best-effort.

3. Rate-limit courtesy: batch field writes per item (all fields for one item together), small sleep only if you hit secondary-rate-limit errors.

## Step 7 — Back-fill dependency references

Dependencies were written as placeholders in Step 5 because issue numbers weren't known yet. Now do a second pass:

For each feature with non-empty `dependencies`:
1. Resolve each dependency feature title to its issue number (from Step 5's map).
2. Replace the Dependencies section in the issue body with real references:
   ```
   ## Dependencies
   - Depends on #<N> — <dependency feature title>
   - Depends on #<N> — <dependency feature title>
   ```
3. Update the issue:
   ```
   gh issue edit <N> --repo <owner/repo> --body-file /tmp/updated-body.md
   ```

If any dependency points to a feature that was skipped as a duplicate of an existing issue, use that existing issue's number — the relationship is still valid.

Skip this step entirely if issue updates aren't available in the toolset; the first-pass placeholders are still human-readable.

## Error handling

- **Missing `project` scope** — stop at the first Project API call that fails with a scope error. Tell the user exactly what to run (`gh auth refresh -s project`) and offer to retry from where you stopped.
- **Rate limits** — if you hit a GitHub rate limit, pause, report how many items have been added + fields set so far, and ask the user whether to resume.
- **Permission denied on a field** — continue without that field; note the missing field in the summary. Other fields on the same item still get written.
- **Permission denied on the project itself** — if `gh project create` fails, check whether an existing project the user can write to is a better fit. If not, **fall back to repo milestones** (old behavior): create one milestone per phase, assign issues to them, and note in the summary that the Project wasn't created.
- **Field-list parsing fails** — without option IDs you can't set single-select values. Skip field-setting for that run and tell the user; the items are still on the board, just uncategorized.

Never retry silently more than once. Surface problems to the user with a concrete next step.
