# Mode: ui-ux

Analyze the application's UI/UX and file concrete improvement issues on GitHub. This mode works best when you can view the app running in a browser (via Puppeteer / a similar tool or screenshots the user attaches). If you can't, fall back to static analysis of component files.

## Categories to cover

1. **usability** — confusing navigation, hidden actions, unclear feedback, poor form UX, missing shortcuts
2. **accessibility** — missing alt text, poor color contrast, keyboard traps, missing ARIA labels, focus management issues
3. **performance** (perceived) — missing loading indicators, slow perceived response, layout shifts, missing skeleton screens, no optimistic updates
4. **visual** — inconsistent spacing, alignment issues, typography hierarchy, color inconsistencies, missing hover/active states
5. **interaction** — missing animations, jarring transitions, no micro-interactions, missing gesture support, poor touch targets

## Analysis process

### If a browser / screenshots are available

1. **Capture the landing state** at desktop width. Look at visual hierarchy, color consistency, typography, spacing, alignment, navigation clarity.
2. **Walk key user flows** — navigation, forms, interactive elements, empty states, loading states.
3. **Check states** — hover, focus, loading, error, success. Capture each.
4. **Mobile viewport** — resize to 375×812. Check navigation, touch targets (min 44×44px), content reflow, readable text sizes.
5. **Accessibility audit** — images without alt, buttons without text, inputs without labels, missing lang attribute, missing page title, color contrast.

### If static analysis only

1. **Component inventory** — identify the component library and design tokens.
2. **Styling patterns** — grep for hover/focus/active states, missing variants.
3. **Accessibility attributes** — count `aria-*`, `role=`, `tabIndex` usage. Flag gaps.
4. **Loading/error/empty states** — grep for `loading`, `isLoading`, `error`, `empty`. Note patterns and missing handlers.
5. **Design consistency** — check for hardcoded colors/spacing that should be tokens.

Note in the issue body whether the finding came from visual inspection or static analysis. The user may want to verify visually before acting.

## Severity (informal)

UI/UX doesn't use a strict severity scale the way security does. Use judgment across three axes:

- **User impact**: how many users hit this, how often?
- **Severity**: how bad is the friction? (accessibility blocker > minor visual inconsistency)
- **Effort**: how hard to fix?

Prioritize accessibility blockers and high-traffic friction first.

## Guidelines

- **Be specific** — "improve buttons" is weak; "add hover state to primary button in `Header.tsx`" is useful
- **Reference components** — point to actual component files
- **Propose concrete changes** — CSS/component diffs, not vague suggestions
- **Match existing patterns** — suggest fixes consistent with the design system already in use
- **Prioritize user impact** — focus on friction that meaningfully affects experience
- **Note screenshots** — if the finding came from a screenshot, include its reference in the issue (or attach it when filing)

## Accessibility checklist (use when auditing)

- [ ] All images have meaningful `alt` (or `alt=""` if decorative)
- [ ] All interactive elements are keyboard-reachable and have visible focus
- [ ] Color contrast meets WCAG AA (4.5:1 for body text, 3:1 for large text)
- [ ] Form inputs have associated labels
- [ ] ARIA landmarks exist (`main`, `nav`, `header`, `footer`)
- [ ] Page has `<html lang="...">` and `<title>`
- [ ] Modals / dialogs trap focus correctly and return it on close
- [ ] Error messages are announced (`aria-live` or equivalent)

## What to skip

- Subjective aesthetic preferences ("I'd use a different color")
- Changes that break the established design system without justification
- Ideas that require a full redesign of a section (file one focused issue per friction point, not mega-issues)
- Duplicates of already-open issues
